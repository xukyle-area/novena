# 推送技术选型

## 核心理由：避免重复设计轮子

> **"内部推送系统选择MQTT主要基于三个考虑：MQTT本身就是完整的消息协议、二进制协议在高频场景更高效、天然适合长连接推送"**

------

## 一、MQTT本身就是完整的消息协议

### MQTT方案：站在标准的肩膀上

```java
// 直接实现RFC 3.1.1标准
// 协议细节已经标准化
public class MqttMessageHandler {
    // MQTT已经定义了完整的消息语义：
    // publish/subscribe、QoS、keepalive、session、offline message
}
```

**MQTT协议现成提供：**
- ✅ 连接管理（CONNECT/CONNACK）
- ✅ 订阅机制（SUBSCRIBE/SUBACK）  
- ✅ 发布语义（PUBLISH/PUBACK）
- ✅ QoS保障（0/1/2级别）
- ✅ 心跳机制（PINGREQ/PINGRESP）
- ✅ 会话管理（Clean Session）
- ✅ 离线消息缓存

------

### WebSocket方案：需要在WebSocket上设计应用层协议

```javascript
// WebSocket更像是一个传输通道，需要业务层补齐消息语义
const applicationProtocol = {
    // 需要自行设计的协议层
    messageTypes: {
        subscribe: { op: "subscribe", channel: "...", symbol: "..." },
        publish: { type: "data", channel: "...", data: "..." },
        ack: { op: "ack", seq: 123 },
        heartbeat: { op: "ping", timestamp: "..." }
    },
    
    // 需要自行实现的机制
    mechanisms: {
        topicRouting: "自定义topic路由逻辑",
        qosHandling: "自定义可靠性保证", 
        sessionManagement: "自定义会话和离线消息",
        reconnectLogic: "自定义断线重连策略"
    }
}
```

**核心问题：本质上是在WebSocket上重新设计一套MQTT**

## 二、二进制协议在高频场景更高效

### 实际数据对比

```text
MQTT PUBLISH消息结构：
固定头(2~5字节) + 可变头(Topic长度标识2字节 + Topic) + Payload

实际示例：
Topic: "BTC" (3字节)
Payload: 价格数据 (8字节)  
总计: ≈ 15~20字节

WebSocket + JSON（业务中常用）:
{"symbol":"BTC","price":67890.12} ≈ 30~40字节

WebSocket + 二进制（需要自定义格式）:
可以接近MQTT大小，但需要自行设计编解码
```

### 真实带宽差异

| 协议方案 | 典型消息大小 | 高频场景优势 |
|---------|------------|------------|
| MQTT | 15~25字节 | 天然二进制 |
| WebSocket+JSON | 30~80字节 | 业务开发简单 |
| WebSocket+二进制 | 15~30字节 | 需要自定义编解码 |

**关键点：MQTT天然是二进制协议，而WebSocket在业务层通常使用JSON**

------

## 三、MQTT天然适合长连接推送

### MQTT的推送优势

MQTT天生就是为**实时推送**设计的：

```text
✅ 长连接管理 - 持续性连接，自动心跳
✅ Topic路由 - 天然的发布订阅模式  
✅ QoS保障 - 0/1/2级别的消息可靠性
✅ 离线消息 - 断线重连后自动补发
✅ 会话保持 - Clean Session控制
```

### WebSocket的推送实现

WebSocket更像是**一个传输通道**，需要业务层补齐：

```text
❌ 连接管理 - 需要自定义心跳和超时
❌ 消息路由 - 需要自定义订阅管理  
❌ 可靠性保证 - 需要自定义ACK机制
❌ 离线处理 - 需要自定义缓存和补发
❌ 状态管理 - 需要自定义会话机制
```

**本质差异：MQTT是消息系统，WebSocket是传输协议**

## 四、选择原因

### 专业版本

> **"我们当时评估过MQTT和WebSocket两种方案，最终内部推送系统选择MQTT，主要有三个原因。"**

### 第一：MQTT本身就是完整的消息协议

MQTT不仅是传输协议，它已经定义了完整的消息语义，包括：

- publish/subscribe  
- QoS
- keepalive
- session  
- offline message

如果使用WebSocket，我们还需要在WebSocket上设计一套应用层协议，比如：

```text
subscribe、publish、ack、heartbeat、reconnect
```

本质上是在重新设计一套消息协议。而MQTT已经标准化（RFC 3.1.1），我们只需要实现协议即可。

### 第二：二进制协议在高频场景更高效

MQTT是紧凑的二进制协议。而WebSocket在实际业务中通常承载JSON。

在高频推送场景：

```text
MQTT payload ≈ 20~40 bytes  
JSON payload ≈ 50~100 bytes
```

在每秒几十万消息时，带宽成本差异非常明显，特别是在移动网络环境下。

### 第三：MQTT天然适合长连接推送

MQTT天生就是为**实时推送**设计：

```text
长连接、topic路由、QoS、离线消息
```

而WebSocket更像是**一个传输通道**，需要业务层自己补齐这些能力。

## 五、常见追问与回应

### Q: 为什么不直接用现成的MQTT Broker？

**A:** "我们使用的是成熟的MQTT Broker（如EMQX），我们实现的是MQTT Gateway/Proxy，而不是完整Broker。"

**架构说明：**
- **MQTT Broker**: 处理连接管理、topic路由、QoS  
- **MQTT Gateway**: 与Kafka集成、业务逻辑处理
- **避免重复造轮子**: 连接管理用成熟组件，业务逻辑做差异化

------

### Q: 高并发场景下MQTT Broker如何支持百万连接？

**A:** "成熟的MQTT Broker（如EMQX）已经解决了这些问题："

**技术要点：**
- **epoll/kqueue**: 高效的IO多路复用
- **Topic Trie**: 高效的主题匹配算法  
- **Session管理**: 内存/持久化混合存储
- **BackPressure**: 慢客户端自动清理机制
- **集群架构**: 水平扩展和负载均衡

------

### Q: 为什么内外选型不同？

**A:** "内部系统追求整体效率，外部接口优先考虑生态兼容性。"

**选型策略：**

| 场景 | 选择 | 核心考虑 |
|------|------|----------|
| **内部系统** | MQTT | 标准协议，避免重复设计 |
| **对外API** | WebSocket | 生态兼容，0成本接入 |
| **移动端推送** | MQTT | 带宽优化，弱网适配 |
| **Web端实时** | WebSocket | 浏览器原生支持 |

------

## 六、技术选型的本质思考

### 核心原则

> **"好的技术选型不是选最酷的技术，而是选最适合解决当前问题的方案"**

**MQTT的价值：**
- 站在RFC标准的肩膀上，专注业务逻辑
- 利用标准化协议带来的生态红利
- 在带宽敏感和IoT场景下的天然优势

**WebSocket的价值：**  
- 最大化接入生态的兼容性
- 降低对外合作的技术门槛  
- 浏览器原生支持，开发成本低

------

## 七、总结

### 选型决策逻辑

```text
内部系统 → 选择MQTT
理由：利用标准协议，避免重新设计轮子

对外接口 → 选择WebSocket  
理由：最大化生态兼容性，降低接入成本

交易所架构 → 内部MQTT + 外部WebSocket
理由：内部效率优先，外部兼容性优先
```

> **核心思维：站在标准的肩膀上，内部追求效率，外部追求兼容**
