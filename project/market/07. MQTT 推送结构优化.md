# 行情推送系统架构优化设计文档

## 1. 背景

当前行情推送系统架构如下：

```
Kafka → Receiver → Dispatcher → Client
```

组件职责：

**Receiver**

* 消费 Kafka 行情消息
* 通过 Netty NIO 推送给 Dispatcher

**Dispatcher**

* 管理客户端 WebSocket / TCP 连接
* 维护订阅关系
* 将行情数据推送给订阅用户

整体架构：

```
Kafka
  │
Receiver
  │
Dispatcher Cluster
  │
Clients
```

---

## 2. 当前架构存在的问题

随着系统规模扩大（更多用户、更多交易对），当前架构存在以下问题：

### 2.1 Receiver 向所有 Dispatcher 广播

Receiver 推送模式：

```
Receiver → Dispatcher1
Receiver → Dispatcher2
Receiver → Dispatcher3
```

所有行情都会发送给 **全部 Dispatcher**。

但实际情况是：

```
Dispatcher1 用户只订阅 BTC
Dispatcher2 用户只订阅 ETH
Dispatcher3 用户只订阅 SOL
```

仍然会产生大量 **无效推送流量**。

---

### 2.2 网络带宽浪费

例如：

```
BTC ticker
```

实际上：

```
只有部分 Dispatcher 需要
```

但当前结构：

```
Receiver → 全部 Dispatcher
```

导致：

```
网络 fanout × Dispatcher 数量
```

---

### 2.3 Receiver 扩容受限

Receiver 当前逻辑：

```
Kafka Consumer
→ Dispatcher Broadcast
```

Receiver 数量增加时：

```
Receiver × Dispatcher
```

连接数增加。

---

## 3. 优化目标

架构优化目标：

### 3.1 支持水平扩容

支持：

```
Receiver Cluster
Push Gateway Cluster
Dispatcher Cluster
```

独立扩容。

---

### 3.2 减少无效推送

目标：

```
行情数据只发送给需要的 Dispatcher
```

避免：

```
Receiver → 全量 Dispatcher
```

广播。

---

### 3.3 消除中心节点

避免出现：

```
单节点 Router
```

成为系统瓶颈。

---

### 3.4 提升整体吞吐能力

支持：

```
百万级 websocket 连接
百万级 msg/s 推送
```

---

# 4. 优化后的系统架构

**Kafka → Receiver Cluster → Push Gateway → Dispatcher → Clients**

优化后架构如下：

```
                    Kafka
                      │
       ┌──────────────┬──────────────┐
       ▼              ▼              ▼
   Partition0     Partition1     Partition2
       │              │              │
       ▼              ▼              ▼
   Receiver1      Receiver2      Receiver3
       │              │              │
       └──────────────┬──────────────┘
                      ▼
               Push Gateway Cluster
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
       Dispatcher  Dispatcher  Dispatcher
          │           │           │
          ▼           ▼           ▼
        Clients     Clients     Clients
```

---

# 5. 核心组件设计

## 5.1 Kafka

Kafka 负责：

```
行情消息总线
```

消息类型包括：

```
ticker
trade
orderbook
kline
```

Topic 示例：

```
market-ticker
market-trade
market-orderbook
```

Partition 策略：

```
key = symbol
```

例如：

```
BTCUSDT → partition0
ETHUSDT → partition1
SOLUSDT → partition2
```

保证：

```
同 symbol 数据顺序
```

---

# 5.2 Receiver Cluster

Receiver 负责：

```
Kafka Consumer
```

职责：

```
1 消费 Kafka 行情数据
2 解码消息
3 推送至 Push Gateway
```

Receiver 特点：

```
无状态
```

只负责：

```
Kafka → 推送
```

Receiver 分片方式：

```
Kafka Consumer Group
```

例如：

```
Receiver1 → partition0
Receiver2 → partition1
Receiver3 → partition2
```

---

# 5.3 Push Gateway

Push Gateway 是本次优化的核心组件。

职责：

```
Symbol Routing
```

功能：

### 5.3.1 Symbol → Dispatcher 映射

维护映射：

```
symbol → dispatcher list
```

例如：

```
BTCUSDT → dispatcher1 dispatcher3
ETHUSDT → dispatcher2
```

当收到：

```
BTC ticker
```

Push Gateway 只发送给：

```
dispatcher1 dispatcher3
```

---

### 5.3.2 动态订阅同步

Dispatcher 会将订阅信息注册到 Push Gateway。

例如：

```
dispatcher1 订阅 BTC
dispatcher2 订阅 ETH
```

Push Gateway 更新路由表。

---

### 5.3.3 Router Cluster

Push Gateway 为集群模式：

```
PushGateway1
PushGateway2
PushGateway3
```

避免：

```
单点瓶颈
```

Receiver 可以连接多个 Gateway。

---

# 5.4 Dispatcher

Dispatcher 负责：

```
客户端连接管理
```

功能：

### 5.4.1 管理客户端连接

例如：

```
WebSocket
TCP
MQTT
```

---

### 5.4.2 管理订阅关系

维护：

```
client → symbol
```

示例：

```
ClientA → BTCUSDT
ClientB → ETHUSDT
```

---

### 5.4.3 向 Push Gateway 注册订阅

当订阅变化时：

```
Dispatcher → Push Gateway
```

更新：

```
symbol routing
```

---

# 6. 数据流说明

完整数据流：

### Step1 行情产生

撮合系统产生行情：

```
BTC ticker
```

写入：

```
Kafka
```

---

### Step2 Receiver 消费

```
Receiver1
```

消费：

```
partition0 (BTC)
```

---

### Step3 Push Gateway Routing

Receiver 推送至 Push Gateway：

```
BTC ticker
```

Push Gateway 查找：

```
symbol routing
```

得到：

```
dispatcher1 dispatcher3
```

---

### Step4 Dispatcher 推送

Dispatcher 推送：

```
client1
client2
```

---

# 7. 系统扩展能力

## 7.1 Receiver 扩容

新增：

```
Receiver4
Receiver5
```

Kafka 自动 rebalance。

---

## 7.2 Push Gateway 扩容

新增：

```
PushGateway4
PushGateway5
```

Receiver 建立新连接。

---

## 7.3 Dispatcher 扩容

新增：

```
Dispatcher 节点
```

通过：

```
负载均衡
```

接入客户端。

---

# 8. 优化效果

优化后系统具备：

### 8.1 减少无效推送

原结构：

```
Receiver → 全 Dispatcher
```

新结构：

```
Receiver → Gateway → 相关 Dispatcher
```

减少：

```
无效 fanout
```

---

### 8.2 网络带宽降低

推送流量减少：

```
50% ~ 90%
```

取决于 symbol 订阅分布。

---

### 8.3 可水平扩展

系统支持独立扩容：

```
Receiver
Push Gateway
Dispatcher
```

---

### 8.4 支持大规模连接

可支持：

```
百万级 websocket
```

---

# 9. 总结

优化后的行情推送系统架构：

```
Kafka
  │
Receiver Cluster
  │
Push Gateway Cluster
  │
Dispatcher Cluster
  │
Clients
```

核心优化点：

```
1 Receiver 无状态
2 Push Gateway Cluster 层
3 Dispatcher 专注连接管理
4 系统可水平扩展
```

该架构能够支持：

```
高并发行情推送
大规模连接
灵活扩容
```

适用于：

```
交易所
行情服务
做市系统
量化行情分发
```
