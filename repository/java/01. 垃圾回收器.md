## 1️⃣ JVM 垃圾回收器分类概览

JVM 常见 GC（HotSpot）可以分为两类：

| 类型                          | 特点            |
| --------------------------- | ------------- |
| 串行/并行收集器（Serial / Parallel） | 停顿时间长，但吞吐量高   |
| 并发收集器（Concurrent）           | 尽量减少停顿时间      |
| 分代收集（Generational）          | 新生代 + 老年代分开回收 |

> CMS 和 G1 都是 **面向大内存、低停顿的 GC**，主要针对老年代对象。

---

## 2️⃣ CMS（Concurrent Mark-Sweep）

**特点：**

* 目标：**尽量减少老年代 GC 停顿**
* **并发标记 + 并发清理**
* 老年代回收器，通常配合 **ParNew** 新生代 GC

### CMS 工作步骤

1. **初始标记（STW）**

   * 停顿，标记从根可达的对象（少量时间）

2. **并发标记**

   * 与应用线程同时运行，标记老年代可达对象

3. **重新标记（STW）**

   * 修正标记过程中因应用线程修改产生的遗漏

4. **并发清理**

   * 回收未被标记的对象

### CMS 优缺点

| 优点     | 缺点                      |
| ------ | ----------------------- |
| 停顿短    | 内存碎片严重，可能触发 **Full GC** |
| 并发执行   | CPU 占用高                 |
| 老年代回收快 | 不压缩空间，碎片累积              |

> CMS 适合 **响应时间要求高、对吞吐量要求一般的应用**。

---

## 3️⃣ G1（Garbage First）

**特点：**

* 面向 **大内存（10GB+）和低停顿应用**
* 逻辑上将堆划分为 **Region**（区域）
* 可以 **同时回收新生代 + 老年代**
* 停顿可预测（可以设置停顿目标，例如 200ms）

### G1 工作机制

1. **堆分区（Region）**

   * 每个 Region 大小固定（1MB-32MB）
   * 新生代、老年代、Humongous 对象都有自己的 Region

2. **回收策略**

   * 统计各个 Region 的垃圾量
   * 优先回收 **垃圾最多的 Region → Garbage First**

3. **混合回收（Mixed GC）**

   * 回收新生代 + 部分老年代 Region
   * 避免 Full GC

4. **并发阶段**

   * 标记阶段并发执行，GC 停顿可控

### G1 优缺点

| 优点         | 缺点                        |
| ---------- | ------------------------- |
| 停顿可预测      | GC 算法复杂，JVM 调优复杂          |
| 自动整理碎片（压缩） | 新生代对象短期内大量分配可能触发 Young GC |
| 避免 Full GC | 对小内存 JVM 并不一定比 CMS 高效     |

> G1 是 **CMS 的升级版**，解决了 CMS 的内存碎片和 Full GC 问题。

---

## 4️⃣ CMS vs G1 对比

| 特性      | CMS           | G1            |
| ------- | ------------- | ------------- |
| 堆划分     | 新生代/老年代       | Region（混合）    |
| 停顿      | 很短，但不压缩       | 可预测，可控制       |
| 内存碎片    | 严重，可能 Full GC | 自动压缩，碎片少      |
| 并发      | 标记 + 清理       | 标记 + 混合回收     |
| 使用场景    | 对延迟敏感，但内存中等   | 大堆内存、低延迟、高吞吐量 |
| Full GC | 可能发生          | 极少发生          |

---

💡 **核心理解：**

* CMS = 并发标记清理，老年代停顿短，但容易碎片 → 可能 Full GC
* G1 = 区域化 + 垃圾优先回收 + 压缩，停顿可控 → 大内存应用首选
