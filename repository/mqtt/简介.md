# 一、MQTT 的核心架构

**MQTT** 是一种**轻量级发布/订阅（Pub/Sub）消息协议**，专门为 **低带宽、不稳定网络、物联网设备** 设计。现在不仅 IoT 在用，很多 **实时推送系统、行情推送、设备控制系统** 也会使用它。

MQTT 采用 **Broker 中心架构**。

```
+-----------+       publish        +-----------+
| Publisher | -------------------> |           |
+-----------+                      |           |
                                   |  Broker   |
+-----------+       subscribe      |           |
| Subscriber| <------------------- |           |
+-----------+                      +-----------+
```

三个核心角色：

### 1 Publisher（发布者）

发送消息的客户端

例如：

* IoT 设备上传数据
* 行情服务器发布价格
* 传感器上传温度

例如：

```
publish topic: sensor/temp
payload: 26.3
```

---

### 2 Subscriber（订阅者）

订阅某个 topic 的客户端。

例如：

```
subscribe: sensor/temp
```

只要有消息发布到这个 topic，就会收到。

---

### 3 Broker（消息服务器）

Broker 是 MQTT 的核心。

常见 Broker：

* EMQX
* Mosquitto
* HiveMQ

Broker 负责：

* 维护连接
* 管理订阅关系
* 转发消息
* QoS 控制
* 持久化（可选）

---

# 二、MQTT 的 Topic 体系

MQTT 通过 **Topic（主题）** 来组织消息。

类似：

```
stock/BTCUSDT/price
stock/BTCUSDT/trade
stock/ETHUSDT/price
```

客户端订阅：

```
stock/BTCUSDT/#
```

`#` 代表 **所有子 topic**

### 支持通配符

#### 1 单层匹配

```
+
```

例如：

```
stock/+/price
```

匹配：

```
stock/BTCUSDT/price
stock/ETHUSDT/price
```

---

#### 2 多层匹配

```
#
```

例如：

```
stock/#
```

匹配：

```
stock/BTCUSDT
stock/BTCUSDT/trade
stock/ETHUSDT/price
```

---

# 三、MQTT 的连接流程

MQTT 建立连接流程：

```
Client -> CONNECT
Broker -> CONNACK
```

连接成功后：

```
Client -> SUBSCRIBE
Broker -> SUBACK
```

发布消息：

```
Client -> PUBLISH
Broker -> 转发给 Subscriber
```

断开：

```
DISCONNECT
```

---

# 四、MQTT QoS（核心机制）

MQTT 最大的特色就是 **QoS（消息质量保证）**

有三个等级。

---

## QoS 0 （最多一次）

```
At most once
```

特点：

* 不保证送达
* 不重试
* 最快

流程：

```
PUBLISH
```

就结束。

适合：

* 行情
* 实时数据
* 不怕丢消息

---

## QoS 1 （至少一次）

```
At least once
```

保证消息至少送达一次。

流程：

```
PUBLISH
PUBACK
```

如果没有收到 ACK：

Broker 会 **重发**

可能导致：

```
重复消息
```

需要客户端 **去重**

---

## QoS 2 （恰好一次）

```
Exactly once
```

最可靠，但最复杂。

流程：

```
PUBLISH
PUBREC
PUBREL
PUBCOMP
```

4次握手。

优点：

* 不丢
* 不重复

缺点：

* 延迟高
* 成本高

---

# 五、MQTT 的连接保持（KeepAlive）

MQTT 是 **长连接协议**。

客户端会定期发送：

```
PINGREQ
```

Broker 回复：

```
PINGRESP
```

如果超时：

Broker 会断开连接。

---

# 六、MQTT 的 Retain 消息

MQTT 支持 **保留消息**。

例如：

```
topic: stock/BTCUSDT/price
retain=true
payload: 63000
```

Broker 会保存最后一条消息。

新订阅者一连接就能收到：

```
63000
```

适合：

* 当前状态
* 最新行情
* 设备状态

---

# 七、MQTT 的 Last Will（遗嘱机制）

MQTT 有个很有意思的机制：

**Last Will and Testament**

如果客户端 **异常断开**：

Broker 会自动发送一条消息。

例如：

```
topic: device/123/status
payload: offline
```

用于：

* 设备掉线检测
* 服务健康检测

---

# 八、MQTT vs HTTP

| 对比  | MQTT    | HTTP             |
| --- | ------- | ---------------- |
| 模型  | Pub/Sub | Request/Response |
| 连接  | 长连接     | 短连接              |
| 带宽  | 非常低     | 较高               |
| 实时性 | 高       | 一般               |
| 推送  | 原生支持    | 需要 websocket     |

所以：

IoT 基本都用 MQTT。

---

# 九、MQTT 在真实系统中的架构

很多系统会这样设计：

```
设备
   │
MQTT Client
   │
MQTT Broker
   │
Kafka / MQ
   │
业务服务
```

例如：

```
IoT设备 -> MQTT -> Broker -> Kafka -> Flink -> 数据处理
```

---

# 十、MQTT 在交易所/行情系统的应用

在一些行情系统中：

```
行情系统
   │
MQTT Broker
   │
WebSocket Gateway
   │
客户端
```

MQTT 的优势：

* 订阅模型
* 高并发连接
* topic routing

但很多交易所最终还是：

```
Kafka -> WebSocket Push
```

因为：

**MQTT 对浏览器不友好**

浏览器通常使用：

```
MQTT over WebSocket
```

---

# 十一、MQTT 为什么这么轻量

MQTT Header 非常小：

```
最小只有 2 bytes
```

而 HTTP：

```
header 常常几百 bytes
```

所以：

IoT 设备特别喜欢。

---

# 十二、MQTT 的一个典型例子

智能家居：

```
灯开关
```

Topic：

```
home/livingroom/light
```

手机控制：

```
publish:
home/livingroom/light = on
```

灯设备：

```
subscribe home/livingroom/light
```

收到后：

```
打开灯
```
