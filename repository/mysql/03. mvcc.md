# MVCC 技术文档（多版本并发控制）

## 1. MVCC 概述

**MVCC（Multi-Version Concurrency Control，多版本并发控制）** 是数据库系统中用于解决**并发读写冲突**的一种机制。

核心思想：

> **通过保存数据的多个历史版本，使读操作无需加锁即可读取历史版本，从而实现读写不阻塞。**

在传统的锁机制中：

| 操作 | 行为   |
| -- | ---- |
| 读  | 加共享锁 |
| 写  | 加排他锁 |

因此会出现：

* 读阻塞写
* 写阻塞读
* 写阻塞写

而 MVCC 的目标是：

| 操作 | 行为     |
| -- | ------ |
| 读  | 读取历史版本 |
| 写  | 生成新版本  |

这样就可以实现：

**读写不冲突**

MVCC 被广泛应用于：

* MySQL InnoDB
* PostgreSQL
* Oracle
* SQL Server (Snapshot Isolation)

---

# 2. 为什么需要 MVCC

数据库并发访问时常见的问题：

## 2.1 脏读（Dirty Read）

事务 A 读取到事务 B **未提交的数据**。

例：

```
T1: update account set money=0
T2: select money -> 0
T1: rollback
```

T2 读取到了不存在的数据。

---

## 2.2 不可重复读（Non-repeatable Read）

同一个事务内，两次读取结果不同。

```
T1: select money -> 100
T2: update money -> 200 commit
T1: select money -> 200
```

---

## 2.3 幻读（Phantom Read）

查询结果行数发生变化。

```
T1: select count(*) where money > 100 -> 10
T2: insert money=200 commit
T1: select count(*) where money > 100 -> 11
```

---

## 3. 传统锁机制的问题

如果只依靠锁：

### 读写冲突

```
T1: select ... for share
T2: update ...
```

读锁会阻塞写锁。

---

### 写写冲突

```
T1: update row
T2: update row
```

必须等待。

---

锁的问题：

* 吞吐量下降
* 死锁
* 阻塞严重

因此数据库引入：

**MVCC**

---

# 4. MVCC 核心思想

MVCC 通过 **版本链 + 快照读** 来实现。

核心原则：

> 每次数据修改都会生成一个新的版本，而不是覆盖旧数据。

例如：

```
时间线：

T1: row = A (version 1)

T2: update row -> B (version 2)

T3: update row -> C (version 3)
```

数据结构：

```
C -> B -> A
```

形成一个 **版本链（Version Chain）**

---

# 5. InnoDB MVCC 实现原理

MySQL InnoDB 的 MVCC 依赖三个隐藏字段：

| 字段          | 含义               |
| ----------- | ---------------- |
| DB_TRX_ID   | 创建该行的事务ID        |
| DB_ROLL_PTR | 回滚指针（指向undo log） |
| DB_ROW_ID   | 行ID（无主键时使用）      |

---

## 5.1 行记录结构

假设一条记录：

```
id = 1
name = "Alice"
```

真实结构：

```
id
name
DB_TRX_ID
DB_ROLL_PTR
```

---

## 5.2 undo log

每次更新都会生成一条 **undo log**：

示例：

```
update name = "Bob"
```

生成：

```
undo log:
name = "Alice"
```

然后形成版本链：

```
Bob (trx=102)
  |
  v
Alice (trx=100)
```

---

# 6. Read View（读视图）

MVCC 的关键：

**Read View**

Read View 决定：

> 当前事务可以看到哪些版本

它包含：

| 字段             | 含义       |
| -------------- | -------- |
| m_ids          | 当前活跃事务列表 |
| min_trx_id     | 最小活跃事务ID |
| max_trx_id     | 下一个事务ID  |
| creator_trx_id | 当前事务ID   |

---

# 7. 可见性判断规则

读取某条数据时：

检查 `DB_TRX_ID`

规则如下：

---

## 7.1 trx_id < min_trx_id

说明事务已经提交。

**可见**

---

## 7.2 trx_id > max_trx_id

说明事务在 ReadView 之后启动。

**不可见**

---

## 7.3 trx_id 在 active list

说明事务还未提交。

**不可见**

---

## 7.4 其他情况

事务已经提交。

**可见**

---

# 8. 快照读 vs 当前读

## 8.1 快照读（Snapshot Read）

普通查询：

```
select * from table
```

特点：

* 使用 MVCC
* 不加锁
* 读取历史版本

---

## 8.2 当前读（Current Read）

例如：

```
select ... for update
select ... lock in share mode
update
delete
insert
```

特点：

* 读取最新版本
* 加锁

---

# 9. 隔离级别与 MVCC

MySQL 默认隔离级别：

```
REPEATABLE READ
```

MVCC 在不同隔离级别下表现不同。

---

## 9.1 READ COMMITTED

每次 SELECT 都生成新的 ReadView。

```
T1 select -> view1
T2 update commit
T1 select -> view2
```

结果不同。

---

## 9.2 REPEATABLE READ

事务第一次 SELECT 时创建 ReadView。

之后复用。

```
T1 select -> view1
T2 update commit
T1 select -> 仍然 view1
```

结果相同。

---

# 10. MVCC 的优点

### 1 高并发

读写不阻塞。

---

### 2 减少锁竞争

大量查询不需要锁。

---

### 3 性能更好

OLTP 场景非常适合。

---

# 11. MVCC 的缺点

### 1 需要存储多个版本

占用空间。

---

### 2 undo log 清理复杂

需要：

```
purge thread
```

清理历史版本。

---

### 3 实现复杂

涉及：

* undo log
* read view
* transaction system

---

# 12. MVCC 工作流程总结

完整流程：

```
事务开始
      │
      │
生成 ReadView
      │
      │
读取数据
      │
      │
检查 DB_TRX_ID
      │
      │
是否可见？
   /       \
 yes       no
  │         │
返回数据   根据ROLL_PTR找旧版本
              │
              │
          直到找到可见版本
```

# 13. MVCC 示例

假设表：

```
user(id,name)
```

初始数据：

```
(1, Alice)
trx_id = 100
```

---

### 事务1

```
T1 start
select name -> Alice
```

生成：

```
ReadView = 100
```

---

### 事务2

```
T2 update name = Bob
commit
trx_id = 101
```

版本链：

```
Bob (101)
  |
Alice (100)
```

---

### 事务1再次读取

```
T1 select
```

检查：

```
101 > readview
```

不可见。

因此读取：

```
Alice
```

---

# 14. MVCC 与锁的关系

MVCC **不能完全替代锁**。

仍然需要：

| 场景   | 机制            |
| ---- | ------------- |
| 快照读  | MVCC          |
| 更新   | 行锁            |
| 防止幻读 | next-key lock |

---

# 15. 总结

MVCC 核心思想：

```
多版本 + 快照读
```

关键组件：

```
undo log
read view
trx id
版本链
```

优势：

```
读写不阻塞
高并发
```

MySQL InnoDB 通过：

```
隐藏字段 + undo log + read view
```

实现了完整的 MVCC 机制。
