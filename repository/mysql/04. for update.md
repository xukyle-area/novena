# 一、SELECT ... FOR UPDATE 是什么

`SELECT ... FOR UPDATE` 是 **当前读（Current Read）**，会对读取到的记录 **加排他锁（X锁）**。

示例：

```sql
BEGIN;

SELECT * FROM user WHERE id = 10 FOR UPDATE;
```

效果：

* 锁住 `id = 10` 这一行
* 其他事务：

  * `UPDATE id=10` → **阻塞**
  * `DELETE id=10` → **阻塞**
  * `SELECT ... FOR UPDATE` → **阻塞**

普通查询：

```sql
SELECT * FROM user WHERE id = 10;
```

不会被阻塞，因为它是 **MVCC快照读**。

---

# 二、什么是间隙锁（Gap Lock）

**间隙锁 = 锁住索引之间的“空隙”**

例如表：

```
id
1
5
10
20
```

存在的间隙：

```
(1,5)
(5,10)
(10,20)
```

如果加了间隙锁：

```
(5,10)
```

那么：

```
INSERT id = 6
INSERT id = 7
INSERT id = 8
```

全部 **被阻塞**。

但：

```
UPDATE id = 5
```

不受影响。

---

# 三、为什么需要间隙锁

间隙锁主要是为了解决 **幻读（Phantom Read）**。

例子：

事务A：

```sql
BEGIN;
SELECT * FROM order WHERE price BETWEEN 10 AND 20 FOR UPDATE;
```

如果没有间隙锁：

事务B可以：

```
INSERT price=15
```

事务A再次查询：

```
SELECT * FROM order WHERE price BETWEEN 10 AND 20;
```

会看到 **新的记录** → 这就是幻读。

所以：

`InnoDB` 会锁住：

```
(10,20) 这个范围
```

阻止插入。

# 四、SELECT FOR UPDATE 什么时候会触发间隙锁

关键看 **查询条件是否命中唯一索引**。

### 1 命中唯一索引

```sql
SELECT * FROM user WHERE id = 10 FOR UPDATE;
```

只锁：

```
id=10 这一行
```

不会有间隙锁。

---

### 2 范围查询

```sql
SELECT * FROM user WHERE id > 10 FOR UPDATE;
```

锁：

```
10之后所有记录 + 间隙
```

也就是：

```
next-key lock
```

---

### 3 非唯一索引

```sql
SELECT * FROM user WHERE age = 20 FOR UPDATE;
```

假设索引：

```
age
18
20
20
25
```

锁的是：

```
(18,20] + (20,25)
```

即：

**Next-Key Lock = 记录锁 + 间隙锁**

---

# 五、三种锁总结

| 锁类型           | 锁什么     |
| ------------- | ------- |
| Record Lock   | 具体行     |
| Gap Lock      | 两行之间的间隙 |
| Next-Key Lock | 行 + 间隙  |

`Next-Key Lock = Record Lock + Gap Lock`

默认情况下：

```
SELECT ... FOR UPDATE
```

使用 **Next-Key Lock**

# 六、一个经典面试题

为什么 `SELECT FOR UPDATE` 会锁住不存在的记录？

例子：

```sql
SELECT * FROM user WHERE id = 7 FOR UPDATE;
```

假设数据：

```
id
5
10
```

会锁住：

```
(5,10)
```

所以：

```
INSERT id=7
```

会被阻塞。

原因：

```
Gap Lock
```

---

# 七、如何避免间隙锁

方法1：使用唯一索引

```
WHERE id = ?
```

方法2：降低隔离级别

```
READ COMMITTED
```

在 `READ COMMITTED` 下：

* **没有 gap lock**
* 只有 **record lock**

---

# 八、一个实际生产问题

交易系统常见死锁：

```
SELECT ... FOR UPDATE
范围查询
```

多个事务锁住不同范围：

```
(10,20)
(20,30)
```

插入：

```
25
15
```

容易 **gap lock 死锁**。

所以金融系统常见策略：

* **避免范围锁**
* 使用 **唯一索引精确锁**
* 或使用 **业务分布式锁**
